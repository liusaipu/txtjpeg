# TxtJPEG 方案文档

## 1. 问题背景

工作目录下的 `result_all_weight_d_正式.txt` 是一个典型的“结果型”ASCII 浮点文本文件：

- **体积**：约 579 MB
- **内容**：约 5789 万个浮点数
- **格式**：单行、超长行、每个数固定宽度 `%9.2f,`（例如 `     0.00,`）
- **特征**：前后段存在大量 0，中段数值动态范围大（最大约 4202）

使用 `zip` / `zstd` 等通用无损压缩，压缩比通常不足 10×，长期存档占用磁盘空间过大。

## 2. 目标

设计并实现一个**有损压缩工具 TxtJPEG**，核心诉求：

1. 显著降低存档体积。
2. 允许用户通过 `quality` 参数在“压缩率”与“保真度”之间权衡。
3. 解压后仍能以接近原文件的格式输出，便于后续查看和近似分析。
4. 不依赖外部数据库，压缩文件自包含。

## 3. 核心方案

### 3.1 基本思路

把 ASCII 浮点序列当作**标量场 / 灰度图像**处理：

```
文本浮点数  →  一维 float32 数组  →  uint8 灰度像素  →  二维图像  →  JPEG 有损压缩  →  .tj 自包含容器
```

### 3.2 为什么选 JPEG

- 文本浮点中大量空白和固定格式属于“表象冗余”，通用文本压缩难以进一步压缩数值本身。
- JPEG 的 DCT 量化可以丢弃人眼/业务不敏感的高频细节，对局部相关的数值矩阵能获得极高压缩比。
- 通过 `quality` 参数可以连续调节有损程度。

### 3.3 为什么用灰度而非 RGB

- 数据本质是**标量序列**，每个值只需一个亮度通道。
- RGB 三通道对单通道数据没有信息增益；强行复制到 R/G/B 还会因 JPEG 色度下采样引入通道间串扰。
- 若业务数据天然为 3 通道，可后续扩展“分通道映射”模式。

### 3.4 有损定位

本方案适合：

- 长期存档，不需要逐字节精确还原。
- 保留整体分布、趋势、量级。
- 后续进行可视化、统计、近似计算。

不适合：

- 需要精确还原的审计/对账场景。
- 对单个数值精度敏感的科学计算。

## 4. 实测效果

在 `result_all_weight_d_正式.txt` 上实测：

| quality | 压缩后体积 | 压缩比 | 最大绝对误差 | 平均绝对误差 | 相对最大误差 |
|---------|-----------|--------|--------------|--------------|--------------|
| 50      | 2.63 MB   | 220×   | 466.82       | 31.24        | 11.1%        |
| 75      | 3.77 MB   | 154×   | —            | —            | —            |
| **85**  | **4.85 MB** | **119×** | **213.33** | **14.53**    | **5.1%**     |
| 95      | 8.66 MB   | 67×    | 116.65       | 8.00         | 2.8%         |

- 原始文件：578.9 MB
- 数值范围：`[0, 4202.2]`
- 解压后文本格式与原文件完全一致（每个 token 宽度、逗号、总长度均相同）

## 5. 推荐用法

```bash
# 默认 quality=85，平衡体积与质量
python txtjpeg.py compress result.txt result.tj --quality 85

# 解压
python txtjpeg.py decompress result.tj result_approx.txt

# 若知道原始矩阵维度，可指定形状以保留局部相关性
python txtjpeg.py compress result.txt result.tj --quality 85 --shape 10000x5789
```

## 6. 下一步可扩展

- **RGB 三通道模式**：对天然 3 通道数据，把三个切片分别映射到 R/G/B。
- **非线性归一化**：对数 / sqrt 映射，提升大动态范围数据的视觉/统计保真度。
- **分块/分层压缩**：对稀疏度差异大的区域使用不同 quality。
- **进度条**：大文件压缩/解压时显示进度。
